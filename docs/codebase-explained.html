Codebase Guide

FootFive is an Express and PostgreSQL backend for a live knockout football tournament simulator.

The application exposes a REST API under /api, streams live simulation events with Server-Sent Events, stores teams, players, fixtures, match reports, odds, and event history in PostgreSQL, and can run a continuous 16-team knockout cup.

Runtime Node.js, Express, CommonJS modules
Database PostgreSQL via pg
Port 9001, API root /api
Realtime SSE endpoint at /api/live/events
Express REST API PostgreSQL persistence Knockout bracket Live match simulation Event replay buffer Jest and Supertest

Runtime Shape

listen.js is both the production entry point and the app export used by tests. It creates the Express app, installs JSON parsing and CORS, mounts the route aggregator under /api, starts listening on port 9001, and optionally starts the live simulation when SIMULATION_AUTO_START=true.

Startup

Running npm start executes node listen.js. If auto-start is enabled, listen.js creates the singleton simulation loop, a singleton event bus, and a new TournamentManager, then calls loop.init(...) and loop.start().

Environment

db/connection.js loads .env.{NODE_ENV} with dotenv, but dotenv does not override existing PG* variables. In this VM, use the documented env -u PGUSER ... prefix for node and npm commands.

Dependency Direction

Normal API requests move from route to controller to model. The simulation path is slightly different: admin or startup code initializes the singleton loop, the loop owns active LiveMatch instances, and the event bus persists and broadcasts events.

HTTP Request Flow

1
Express receives the request. listen.js parses JSON, applies CORS, and sends anything under /api to routes/index.js.
2
The route module chooses a controller. Example: GET /api/fixtures/:id/events calls getMatchEvents in fixtureController.js.
3
The controller validates and orchestrates. Controllers parse params and query strings, enforce simple validation, call models or simulation singletons, and shape JSON responses.
4
Models execute SQL. Models use parameterized queries through db/connection.js. Some bulk inserts use pg-format.
5
The controller returns JSON or SSE. REST endpoints return JSON. /api/live/events keeps the response open and writes SSE messages.

API Surface

The API base is http://localhost:9001/api in development. Public read endpoints are open. Admin endpoints are open outside production and require DEV_ADMIN=true or a matching x-admin-secret header in production.

/api GET / routes/index.js Smoke response: { msg: "ok" }.
/api/teams GET /, GET /3jcup, GET /stats teamController.js Team list with ratings, top JCup standings, and aggregate team stats.
/api/players GET /, GET /team/:teamName, GET /:playerId playerController.js Player list, player by ID, and players by team name.
/api/fixtures POST /, POST /batch, GET /, GET /:id, DELETE /:id fixtureController.js Create, list, inspect, and delete fixtures. Listing supports filters such as status, teamId, round, and tournamentId.
/api/fixtures GET /:id/odds, POST /:id/odds/calculate, POST /:id/simulate fixtureController.js Retrieve or recalculate odds. The simulate route uses the deprecated synchronous SimulationEngine.
/api/fixtures GET /:id/report, GET /:id/events, GET /:id/goals fixtureController.js Post-match statistics, persisted match events, and goal-only event history.
/api/live GET /events, GET /status, GET /tournament, GET /matches, GET /matches/:fixtureId, GET /fixtures, GET /events/recent liveController.js SSE stream, current loop status, current tournament snapshot, active match snapshots, enriched fixture list, and recent event replay.
/api/admin POST /simulation/start, POST /simulation/stop, POST /tournament/start, POST /tournament/cancel, POST /tournament/skip-to-round, POST /clock/*, POST /match/:fixtureId/*, GET /state adminController.js Start, stop, pause, resume, speed up, inspect, skip, force-score, and force-end the live simulator.
/api/diagnostic GET /, POST /seed diagnosticController.js Database status, table counts, sample rows, and manual seed trigger.

Database Layer

db/connection.js exports a PostgreSQL pool. Every model imports that pool and performs SQL directly. The schema is created by db/seed.js for a fresh database and extended through SQL migrations in db/migrations/.

Table Owned by Purpose Important fields
teams TeamModel Team identity and long-term tournament statistics. wins, losses, goals_for, goals_against, jcups_won, runner_ups, recent_form, goal_diff.
players PlayerModel Players attached to teams, including the goalkeeper marker. attack, defense, is_goalkeeper.
fixtures FixtureModel Scheduled, live, and completed matches, including bracket slots. tournament_id, round, status, scores, penalties, winner_team_id, bracket_slot, feeds_into.
match_events MatchEventModel Persisted timeline events for each fixture. event_type, minute, second, team_id, player_id, bundle_id, metadata, seq, server_timestamp.
match_reports MatchReportModel Aggregate post-match statistics. Possession, shots, xG, corners, fouls, cards, extra-time flag, penalties flag.
fixture_odds OddsModel Pre-match probabilities and decimal odds. Home and away probabilities, odds, margin, JSON factors.
tournament_state Migration only Planned/persisted tournament lifecycle state. state, current_round, timing fields, and total_match_minutes.

Model Responsibilities

TeamModel Builds team DTOs with derived ratings from player rows, updates wins/losses/goals, JCup wins, runner-up counts, highest round, and stats tables.
FixtureModel Creates fixtures with nullable teams for TBD bracket slots, filters fixture lists, completes matches, and fills next-round home or away slots.
MatchEventModel Inserts single or batch events, retrieves by fixture/type/bundle, supports live catch-up by event ID, and formats events for API responses.
MatchReportModel Stores final stat lines and returns combined fixture/report details for report endpoints.
OddsModel Creates, updates, upserts, deletes, and formats odds rows.
UserModel Placeholder for future authentication work. It currently exports an empty object.

Seeding and Migrations

npm run seed executes db/run-seed.js, which drops and recreates the core tables, creates indexes and a goal_diff trigger, then inserts team/player seed data. Migrations add or alter match system features over time, including bracket fields, tournament state, expanded event types, sequence fields, and chain narrative event types.

Simulation System

The current live simulator lives in gamelogic/simulation/. Its central design is a singleton SimulationLoop that ticks once per second, delegates tournament lifecycle work to TournamentManager, and ticks each active LiveMatch. 

Admin or auto-start
  -> getSimulationLoop()
  -> getEventBus()
  -> new TournamentManager(rules)
  -> loop.init({ tournamentManager, eventBus })
  -> loop.start()
  -> loop.tick() every second
      -> tournamentManager.tick(now)
      -> liveMatch.tick(now)
      -> eventBus.emit(event)
      -> tournamentManager.onMatchFinalized(result)

SimulationLoop

SimulationLoop.js is responsible for process-local simulation coordination. It stores the active matches in a Map, exposes admin controls such as pause/resume/speed/force-end/force-score, attempts recovery on start, and emits tournament-level events through the event bus when rounds and fixtures change.

TournamentManager

TournamentManager.js owns the tournament state machine. It loads teams, creates a tournament ID, shuffles teams, pre-generates the full bracket, starts each round, collects winners, updates team history, records the final winner/runner-up, and supports recovery from recent fixtures after restart.

State group States Meaning
Idle/setup IDLE, SETUP No active tournament, or teams and bracket fixtures are being prepared.
Playing rounds ROUND_OF_16, QUARTER_FINALS, SEMI_FINALS, FINAL Live matches are created and ticked for the current round.
Breaks QF_BREAK, SF_BREAK, FINAL_BREAK, TOURNAMENT_BREAK Timer-driven pauses between rounds or between tournaments in continuous mode.
Results RESULTS, COMPLETE Winner and runner-up are determined, team stats are updated, and state is cleared for the next tournament.

BracketManager

BracketManager.js creates all 15 bracket fixtures at tournament setup time. Round of 16 fixtures are assigned teams immediately. Quarter-final, semi-final, and final fixtures start with nullable team IDs and are filled as winners advance through bracket_slot and feeds_into. 

R16_1 -> QF1 home      R16_2 -> QF1 away
R16_3 -> QF2 home      R16_4 -> QF2 away
R16_5 -> QF3 home      R16_6 -> QF3 away
R16_7 -> QF4 home      R16_8 -> QF4 away
QF1   -> SF1 home      QF2   -> SF1 away
QF3   -> SF2 home      QF4   -> SF2 away
SF1   -> FINAL home    SF2   -> FINAL away

LiveMatch

LiveMatch.js simulates one fixture. It maps real ticks to match minutes, transitions through match phases, loads players, delegates event creation to EventGenerator, delegates shootouts to PenaltyShootout, updates fixture scores during play, finalizes the fixture, writes the match report, updates team statistics, and advances winners into the bracket.

Match state Default tick range Match minutes
FIRST_HALF 0 to 239 seconds 1 to 45
HALFTIME 240 to 299 seconds 45
SECOND_HALF 300 to 539 seconds 46 to 90
EXTRA_TIME_1, ET_HALFTIME, EXTRA_TIME_2 After 90 minutes if a knockout match is drawn 91 to 120
PENALTIES After extra time if still tied 120
FINISHED After winner is known Final minute

EventGenerator and PenaltyShootout

EventGenerator handles minute-level play simulation: possession, fouls, cards, attacks, penalties, shots, saves, misses, goals, xG, momentum, pressure, and narrative payloads. PenaltyShootout handles staged walkups, outcomes, reactions, standard five-round shootouts, sudden death, and instant shootouts for forced endings.

Legacy SimulationEngine

SimulationEngine.js is marked deprecated and only backs POST /api/fixtures/:id/simulate. It runs a whole fixture synchronously and persists the result. New live tournament behavior should be implemented in LiveMatch, EventGenerator, PenaltyShootout, TournamentManager, or adjacent simulation modules.

Events and SSE

EventBus.js is the central live event hub. It accepts raw events from matches and the tournament manager, normalizes them, assigns a monotonically increasing seq, stamps serverTimestamp, categorizes by type, buffers recent events, persists eligible match events, and writes matching events to connected SSE clients. 

LiveMatch or TournamentManager event
  -> EventBus._normalizeEvent()
  -> seq and serverTimestamp assigned
  -> buffered in memory
  -> persisted if scope=match and type is persistable
  -> broadcast to SSE clients whose filters match
  -> emitted internally through EventEmitter

SSE Client Flow

GET /api/live/events accepts filters: tournamentId, fixtureId, category, and afterSeq. The event bus stores each client response in a map, sends a connected event, then writes events as: 

event: goal
data: {"seq":12,"scope":"match","fixtureId":42,"type":"goal",...}

Event Shape

Field Meaning
scope match, tournament, or system.
tournamentId Current tournament identifier when available.
fixtureId Fixture identifier for match-scoped events.
minute Match minute for match-scoped events. Tournament/system events use null.
type Canonical event type from gamelogic/constants.js.
category Array of categories such as flow, highlights, goals, cards, shootout, tournament, or result.
payload Event-specific data. The bus folds unknown top-level raw event keys into this object.
seq In-memory event sequence used for ordering and reconnect catch-up.
serverTimestamp Epoch milliseconds assigned by the event bus.

Persistence Rules

Only events with scope === "match", a non-null fixtureId, and a type included in PERSISTABLE_MATCH_EVENT_TYPES are inserted into match_events. Persistence is fire-and-forget: a database error is logged but does not crash the live simulation.

Odds Engine

OddsEngine.js calculates two-outcome decimal odds for fixtures. It combines recent form, team attack/defense ratings, goal difference, goalkeeper rating, and a small home advantage, then applies a configurable margin. The default controller and service instantiate it with a 0.05 margin.

Inputs Team rating DTOs from Team.getRatingById and recent form from Fixture.getTeamForm.
Outputs homeWinProb, awayWinProb, decimal odds, margin, and JSON factors explaining the calculation.

Tests

Jest is configured in jest.config.js. Tests are split into unit tests, integration tests, and API route tests. Supertest is used for route-level checks. Database-backed tests use setup helpers to seed and clean the test database.

Command Scope
npm test All Jest tests.
npm run test:unit Unit tests for controllers, models, and simulation modules.
npm run test:integration Integration tests against controller/database behavior and event persistence.
npm run test:api Route-level API tests through Express.
npm run test:coverage Coverage report for models, controllers, routes, and game logic.

Required command prefix in this VM

Because the VM pre-injects PostgreSQL environment variables, run tests with: 

env -u PGUSER -u PGDATABASE -u PGPASSWORD -u PGHOST -u PGPORT npm test

__tests__/setup/testHelpers.js provides getTestApp(), createTestApp(), database setup/cleanup helpers, team/player factories, and a small SSE client for event stream tests. db/test-connection.js enforces that the test database is named footfive_test.

Watch-outs

PG env variables

The biggest local execution trap is pre-existing PG* environment variables. Use the env -u prefix for node and npm commands in this VM.

Seed and migrations must match

Freshly seeded databases must have the same effective schema as migrated databases. This matters most for match_events.event_type CHECK constraints and event metadata columns.

Controller/service split is incomplete

services/ contains extracted logic, but several controllers still implement equivalent logic directly. When refactoring, migrate behavior carefully and keep API responses stable.

Simulation state is process-local

SimulationLoop and EventBus are in-memory singletons. Horizontal scaling would require shared state and shared event distribution, such as Redis or another coordination service.

Route names and comments can drift

Verify behavior against route and controller source. Example: the /api/teams/3jcup path and "top 3" naming do not line up cleanly with the model limit of 16.

Export barrels need checks

gamelogic/simulation/index.js exports several names from TournamentManager. Confirm those names are actually exported before depending on that barrel in new code.
